### 6. `app.py` — Flask backend (the most important file in the audit)

**What it does:** Loads the trained model via three fallback strategies, exposes `/` (serves the SPA) and `POST /predict` (image → class probabilities + medical-style description/recommendation).

| Severity | Issue | Location | Explanation | Fix |
|---|---|---|---|---|
| 🔴 **MUST FIX** | **`debug=True` in production entrypoint** | `app.run(host="0.0.0.0", port=5000, debug=True)` | Flask's debug mode enables the Werkzeug interactive debugger. Combined with `host="0.0.0.0"` (binds every network interface), **any unauthenticated visitor who triggers an unhandled exception can get a debugger console with arbitrary code execution** on your server. This is a critical RCE risk the moment this is deployed anywhere reachable. | ```python\nimport os\nDEBUG = os.environ.get("FLASK_DEBUG", "0") == "1"\nif __name__ == "__main__":\n    app.run(host="0.0.0.0", port=int(os.environ.get("PORT", 5000)), debug=DEBUG)\n``` Never set `FLASK_DEBUG=1` outside local development. |
| 🔴 **MUST FIX** | **Broken model path logic** | `MODEL_PATH = "skin_disease_v2.keras"`<br>`WEIGHTS_PATH = os.path.join(MODEL_PATH, "model.weights.h5")` | `os.path.join` treats `MODEL_PATH` as a *directory*, producing `"skin_disease_v2.keras/model.weights.h5"` and `"skin_disease_v2.keras/config.json"` — paths that don't exist anywhere in the supplied project (the real files are flat: `config.json`, `model_weights.h5`). Strategy 1 (and likely Strategy 2, since `skin_disease_v2.keras` itself isn't in the upload) will fail every time, silently falling through to the legacy `.h5`. The app currently "works" only by accident, via Strategy 3. | Use real, explicit, flat paths: ```python\nCONFIG_PATH = "config.json"\nWEIGHTS_PATH = "model_weights.h5"\nLEGACY_H5_PATH = "skin_disease_mobilenet.h5"\n``` and load in that order, removing the broken `os.path.join` entirely. |
| 🔴 **MUST FIX** | **Unsafe deserialization (`safe_mode=False`)** | `tf.keras.models.load_model(MODEL_PATH, compile=False, safe_mode=False)` | `safe_mode=False` explicitly disables Keras 3's protection against arbitrary Python code execution during model deserialization (e.g. via `Lambda` layers or pickled config). This is safe **only** if you 100% control and trust the model file's provenance. Publishing this pattern in an open-source repo invites copy-paste misuse against untrusted/downloaded models — a real deserialization vulnerability class. | Drop `safe_mode=False` (default `True`) unless a specific layer genuinely requires it, and if it does, document *why* prominently and keep the model file's origin verifiable (e.g., checksum it in CI). |
| 🔴 **MUST FIX** | **Internal error details leaked to clients** | `return jsonify({"detail": f"Server processing error: {str(error)}"}), 500` | Returns raw exception text (which can include file paths, library internals, stack details) directly in the HTTP response — an information-disclosure risk. | Log the full exception server-side (`logging.exception(...)`) and return a generic message: `return jsonify({"detail": "Server processing error."}), 500`. |
| 🟠 SHOULD FIX | **Wide-open CORS** | `CORS(app)` | With no `origins=` restriction, `flask-cors` defaults to `Access-Control-Allow-Origin: *`, letting *any* website call your inference API directly from a user's browser. Fine for a demo, risky if you don't want third parties embedding/scraping your model. | `CORS(app, origins=os.environ.get("ALLOWED_ORIGINS", "http://localhost:5000").split(","))` |
| 🟠 SHOULD FIX | **No upload size limit at the Flask level** | none set | Client-side JS caps uploads at 10 MB, but that's trivially bypassed (curl/Postman/modified JS). Without a server-side cap, a large/malicious payload can be fully buffered into memory before any check runs. | `app.config["MAX_CONTENT_LENGTH"] = 10 * 1024 * 1024` |
| 🟠 SHOULD FIX | **No server-side file-type validation beyond "can PIL open it"** | `predict()` | Relies entirely on `Image.open` failing for non-images; doesn't check MIME/extension server-side at all (client-side check is cosmetic). | Validate `file.mimetype` and/or use `Pillow`'s `Image.verify()` before full decode; reject non-allow-listed types explicitly. |
| 🟠 SHOULD FIX | **`print()`-based logging** | throughout | Hard to filter/route in production, no log levels, no timestamps by default. | Replace with Python's `logging` module (`logging.basicConfig(level=logging.INFO)`, `logger.info/warning/error`). |
| 🟠 SHOULD FIX | **No rate limiting / no auth on a public inference endpoint** | `/predict` | Anyone can hit the endpoint repeatedly, consuming model compute. Not a vulnerability per se, but a denial-of-service / cost-abuse risk once deployed publicly. | Add `Flask-Limiter` (e.g. 10 requests/minute/IP) for any public deployment. |
| 🟠 SHOULD FIX | **Monolithic single-file design** | whole file | Config, medical-content data (`CLASS_INFO`), model-loading logic, and routes are all mixed in one file — fine for a demo, not great for maintainability/scalability. | Split into `config.py`, `data/class_info.json`, `services/model_loader.py`, `routes/predict.py`, with `app.py` reduced to app factory + route registration. |
| 🟢 OK | Good: presence/empty-filename checks on upload, try/except around inference with a structured JSON error contract, clean separation of `CLASS_INFO` from prediction logic, sensible probability rounding. | — | — |

## Cross-Cutting ML Best-Practice Notes

- **Train/val/test discipline:** Only train/val are used during training (per the docx); no independent test set evaluation is shown. Add one before claiming any accuracy figure publicly.
- **Reproducibility:** No seeding, no logged library versions beyond `keras_version` in `metadata.json`, no requirements pinning anywhere in the provided files. Fix via `requirements.txt` (provided below) + a fixed seed + a training config file/log.
- **Data leakage:** No direct evidence of train/val contamination in the code shown, but the dataset split itself (`Split_smol/train` vs `Split_smol/val`) happens *outside* any of the provided files — document exactly how/where that split was created so reviewers (and future-you) can verify there's no overlap (e.g., same source image augmented into both sets).
- **Label integrity:** The class-name mismatch risk (docx Cell 7 vs Cell 8, and `class_names.json` being hand-typed) is the single most important reproducibility bug to close before calling this "production".

# Final Security Report

**Security Score: 58 / 100**

**Risk assessment:** No leaked credentials, API keys, or personal data were found embedded in code or model binaries — that's the good news. The score is held down primarily by three **deployment-breaking-or-dangerous** issues in `app.py` (`debug=True` + `0.0.0.0`, `safe_mode=False` model deserialization, and raw exception leakage to clients), plus one **PII leak** in the training-instructions document (a Windows local username/path). None of these require deep expertise to fix, but all three `app.py` issues are the kind that get exploited automatically by scanners within minutes of a debug Flask server going live publicly.

**Vulnerabilities found:**
1. 🔴 Flask debug mode + open host bind → potential remote code execution if deployed as-is.
2. 🔴 `safe_mode=False` Keras model loading → unsafe deserialization pattern.
3. 🔴 Raw exception text returned to API clients → information disclosure.
4. 🔴 Personal local file path (Windows username) in `colabb_codee.docx`.
5. 🟠 Unrestricted CORS (`*`) on a public inference API.
6. 🟠 No server-side upload size/type enforcement.
7. 🟠 Zip extraction without path-traversal guard in the training notebook.

**Fixes:** see the per-file tables above; all are mechanical and low-effort.
